## 1. 产品愿景 / Product Vision

### 1.1 一句话定义

> **axon.ai 是一个面向开发者的 AI harness 框架:它理解你的开发意图,在隔离的虚拟机中自主完成编码任务,并随着协作越用越懂你。**

### 1.2 愿景陈述 / Vision Statement

把"AI 辅助编码"从**单轮问答**升级为**可编排、可隔离、可记忆的多智能体开发系统**。开发者不再逐行指挥 AI,而是下达目标,由框架拆解任务、调度隔离环境执行、自检验收,沉淀长期记忆形成个性化协作。

### 1.3 价值主张 / Value Proposition

| 维度 | 现状痛点 | axon.ai 提供 |
|------|---------|-------------|
| **隔离** | AI agent 直接在主机改代码,失败/误操作污染环境 | 每个任务在独立 microVM 中执行,失败即丢弃 |
| **记忆** | 每次对话从零开始,要反复交代项目背景与个人偏好 | 长期记忆沉淀用户画像与项目知识,越用越懂 |
| **编排** | 单个 AI 一次只做一件事,复杂任务要人工拆解 | 调度大脑自动拆解为 DAG,多 agent 并行执行 |
| **可控** | AI 行为黑盒,难以干预 | 记忆可调节、任务可中断、验收标准显式 |

## 2. 目标用户与场景 / Target Users & Scenarios

### 2.1 目标用户

**主要用户**:个人开发者 / 小型开发团队,具备一定工程能力,希望用 AI 提升开发效率但不满于现有 Copilot 类工具的"单轮辅助"模式。

**次级用户**:AI 应用研究者,需要一个可实验多 agent 编排与记忆系统的框架。

### 2.2 核心场景 / Core Scenarios

**场景 A — 功能开发**
> 用户:"帮我实现一个用户登录接口,用 JWT,要写测试。"
> axon.ai:调度大脑拆解(设计 schema → 写接口 → 写测试 → 自检)→ 在隔离 VM 中执行 → 返回可运行代码 + 测试通过报告。

**场景 B — 重构**
> 用户:"把这个 800 行的函数按职责拆分。"
> axon.ai:分析依赖 → 拆分子任务 → 多 VM 并行重构不同模块 → 合并验证编译。

**场景 C — 长期协作**
> 用户多次让 axon.ai 改同一项目。系统记住"该用户偏好用 `thiserror` 而非 `anyhow` 做库错误""该项目测试用 `testcontainers`",后续任务自动遵循,无需重复交代。

**场景 D — 记忆管理**
> 用户发现 axon.ai 记错了一条偏好,通过 CLI / Web 查看记忆列表,删除错误条目、调节权重。

## 5. 用户故事 / User Stories

选取核心 Story 展开为标准格式(As a / I want / So that + 验收标准)。

### US-001 提交开发任务
> **作为**开发者,**我想**用一行命令提交开发目标,**以便**让框架自动拆解并执行。

**验收标准**:
- `axon run --goal "..."` 返回任务 ID
- 任务进入队列,状态可查
- (M1)至少完成单步任务执行

### US-002 查看任务进度
> **作为**开发者,**我想**查看任务状态与执行日志,**以便**了解进度并在异常时干预。

**验收标准**:
- `axon tasks` 列出所有任务及状态
- 可查看单任务执行日志
- 状态机:Queued→Running→Completed/Failed

### US-003 隔离执行不污染主机
> **作为**开发者,**我想**任务在隔离环境执行,**以便**失败的实验不影响我的主机。

**验收标准**:
- 每个任务在独立容器/VM 中执行
- 失败任务的环境可丢弃,主机无残留
- 工作目录改动经显式挂载,可控

### US-004 系统记住我的偏好
> **作为**开发者,**我想**系统记住我的开发习惯,**以便**不用每次重复交代。

**验收标准**:
- 跨会话保留偏好(库选择、命名风格、测试约定)
- 新任务自动应用记忆
- 偏好错误时可修正

### US-005 管理与纠正记忆
> **作为**开发者,**我想**查看并调节系统记忆,**以便**纠正错误、控制上下文。

**验收标准**:
- 列出所有记忆(可按类别过滤)
- 删除错误记忆
- 调节记忆权重(影响召回排序)

### US-006 验收闭环
> **作为**开发者,**我想**系统自检任务产物再交付,**以便**拿到的是可运行的代码。

**验收标准**:
- worker 执行后运行测试/构建自检
- 自检失败自动重试(有限次)
- 最终交付附自检报告

## 6. 优先级矩阵 / Priority Matrix

采用 **MoSCoW** 优先级法,结合里程碑:

| 优先级 | 含义 | 包含 |
|--------|------|------|
| **M (Must)** | 必须有,无则产品不成立 | M0-M1 全部:LLM 接入、单步任务、Docker 隔离、CLI、自检 |
| **S (Should)** | 应该有,核心差异化 | M2 记忆系统、M3 Firecracker 强隔离 |
| **C (Could)** | 可以有,增强体验 | M4 Web 仪表盘、分布式、可观测性 |
| **W (Won't)** | 本期不做 | 多租户、GUI IDE 插件、自有大模型训练 |

### Kano 维度补充

| 功能 | 基本/期望/兴奋 |
|------|---------------|
| 任务隔离执行 | 基本(必须) |
| 记忆系统 | 兴奋(差异化) |
| Web 仪表盘 | 期望 |
| 多 agent 协作 | 兴奋 |
| 成本监控 | 期望 |

## 7. 版本规划 / Release Planning

版本与里程碑对应,每个版本交付**可用的用户价值**(不只是技术内部达成)。

### v0.1 — M0 骨架 ✅ 已完成
- **用户价值**:无(内部基础设施)
- **交付**:workspace + trait 占位 + 技术选型文档
- **状态**:已完成并提交

### v0.2 — M1 单机可用 (Single-Node MVP)
- **用户价值**:开发者可用 CLI 提交单步开发任务,在 Docker 隔离中执行并自检
- **关键 Stories**:US-001、US-002、US-003、US-006
- **验收**:端到端跑通 `axon run` → Docker 执行 → 自检 → 回报
- **依赖**:OpenAI/DeepSeek API key、本地 Docker

### v0.3 — M2 记忆系统
- **用户价值**:系统跨会话记住用户偏好与项目知识,可管理
- **关键 Stories**:US-004、US-005
- **验收**:跨会话应用记忆 + CLI 记忆管理命令可用
- **依赖**:Qdrant、redb

### v0.4 — M3 强隔离
- **用户价值**:任务在 Firecracker microVM 中执行,生产级隔离
- **关键 Stories**:S3.4.x(VM 生命周期/资源限额/快照)
- **验收**:Linux 上 Firecracker 集成测试通过
- **依赖**:Linux + KVM

### v0.5 — M4 分布式 + Web
- **用户价值**:多节点调度 + Web 仪表盘可视化监控
- **关键 Stories**:S3.1.3、S5.2.x、S6.2.1
- **验收**:多 worker 节点 + Web 实时监控
- **依赖**:NATS

### v0.6+ — M5 高级特性
- **用户价值**:多步 DAG 编排、多 agent 协作、成本监控
- **关键 Stories**:S1.2.2、S1.4.1、S3.5.1、S6.4.1
- **验收**:复杂任务自动 DAG 拆解并行执行

## 8. 非功能性需求 / Non-Functional Requirements

| 类别 | 要求 |
|------|------|
| **性能** | 单步任务端到端 < 60s(不含 LLM 响应);VM 启动 < 2s(Firecracker 快照 < 500ms) |
| **隔离安全** | 任务间零共享;失败 VM 资源完全释放;无主机文件残留 |
| **可靠性** | 任务失败可重试;调度器崩溃不丢队列状态(持久化) |
| **可观测** | 全链路 tracing;任务事件可订阅;LLM 成本可统计 |
| **可配置** | LLM provider、隔离后端、并发度、记忆策略均可配置 |
| **可移植** | 开发期跨平台(Docker);生产 Linux(KVM) |
| **成本** | 支持本地模型 fallback 降本;请求缓存 |

## 9. 范围边界 / Out of Scope

本期(至 v0.6)**明确不做**:

- ❌ 多租户 / 用户认证授权(单用户/单团队场景)
- ❌ IDE 插件(VSCode / JetBrains 集成)
- ❌ 自有大模型训练(只接入现有 provider)
- ❌ 图形化 IDE 式界面(仅 Web 仪表盘)
- ❌ 移动端
- ❌ 任务 marketplace / agent 商店

这些可能作为未来方向,但不纳入当前路线。
